# /inbox-process — сортировка инбокса

Разбирает необработанные заметки из `__BRAIN_DIR__/inbox/*.md` и раскладывает их по brain согласно `__BRAIN_DIR__/CONVENTIONS.md`. Обязательно прочитай CONVENTIONS.md перед началом — особенно разделы «Общие правила» и «Правила сортировщика».

## Шаги

### 1. Собрать вход

- Файлы `__BRAIN_DIR__/inbox/*.md` (не `archive/`, не `sessions*.log`). Пусто → ответь «инбокс пуст» и закончи.
- Прочитай `__BRAIN_DIR__/INDEX.md` — карта существующих сущностей. Маршрутизация идёт **в существующие** сущности; новая создаётся только если ничего не подошло, с явной пометкой в отчёте.

### 2. Классифицировать каждую заметку

Категории:

| Категория | Куда | Признак |
|---|---|---|
| задача | `tasks/<task>/journal.md` (append) | упомянута текущая работа / именованная активность |
| проект | `projects/<id>/` | относится к проекту из INDEX |
| знание | `knowledge/<topic>.md` | подтверждённый факт, не привязанный к задаче |
| урок/правило | `rules/<topic>.md` | «надо всегда/никогда...», коррекция, грабли |
| паттерн | `learning/patterns.md` | «опять делаю X руками», повторяющееся действие |
| петля (хвост) | `loops.md` (секция Сделать/Жду/Вопросы/Потом) | «не забыть», «надо будет», «жду от», «спросить у» — незавершённое вне задач |
| TODO/внешнее действие | предложить пользователю | «написать Y», «отправить Z» |
| мусор | archive без маршрута | потерявшее актуальность, дубль |

Одна заметка может уйти в несколько мест — оба маршрута в отчёт.

### 3. Применить правила записи (из CONVENTIONS.md)

- **Источник**: frontmatter `source:` или строка-пометка. Источник заметки = «пользователь, <дата заметки>».
- **Дедуп**: перед созданием записи — поиск существующей (INDEX + grep по knowledge/rules). Есть → обновить её. Противоречие → пометить конфликт, старое → `status: outdated`, не перезаписывать молча.
- **Гипотезы в knowledge не попадают** — только подтверждённые факты (`status: verified`). Неподтверждённое → journal соответствующей задачи с маркером `гипотеза:`.
- **Абсолютные даты**: «завтра» в заметке от 2026-06-04 → «2026-06-05».
- **Секреты** → `[secret removed]`.
- Язык пользователя, kebab-case имена файлов, `[[wikilinks]]` на связанные сущности.

### 4. Неоднозначности

Не угадывать. Копи неоднозначные заметки по ходу и задай **одной пачкой** через AskUserQuestion в конце классификации, до записи. Варианты — конкретные маршруты, не абстракции.

### 5. Внешние действия

Заметки типа «написать X», «отправить Y» — НЕ выполнять. Включи в отчёт секцией «Требуют действия» с предложением. Выполнять только после явного согласия пользователя ([[rules/external-actions-confirm]]).

### 6. Записать и заархивировать

1. Внеси записи по маршрутам.
2. Обнови `INDEX.md` (новые сущности — одной строкой).
3. Перенеси обработанные файлы в `inbox/archive/`, дописав в конец каждого блок:

   ```markdown
   ---
   processed: <YYYY-MM-DD>
   routing:
   - "<первые слова заметки>" → <путь назначения>
   ```

4. Закоммить brain и запушить: `cd __BRAIN_DIR__ && git add -A && git commit -m "inbox-process: <дата>" && git push` (пуш — если настроен remote; brain не git-репозиторий → пропустить; [[rules/brain-backup]]).

### 7. Отчёт

Таблица: заметка (кратко) → куда → почему. Отдельно: созданные новые сущности (с обоснованием), конфликты, секция «Требуют действия».

## Запреты

- Не удалять заметки — только archive.
- Не выполнять внешние действия (письма, сообщения, что-либо вне brain) без подтверждения.
- Не плодить категории: сомневаешься между существующей и новой — выбирай существующую или спрашивай.
