# /retro — ретроспектива поперёк сессий

Анализирует транскрипты за период и отвечает на вопрос «**как** работает агент», а не «что делалось» (это `/digest`). Два контура: майнинг паттернов и аудит обучения. Правила — `__BRAIN_DIR__/CONVENTIONS.md`, разделы «learning/» и «Контур обучения».

## Файлы

- `~/.claude/projects/*/` — транскрипты сессий (`*.jsonl`)
- `__BRAIN_DIR__/learning/patterns.md` — наблюдения со счётчиками (пополняет retro и digest)
- `__BRAIN_DIR__/learning/proposals.md` — предложения, достигшие порога ≥3 наблюдений
- `__BRAIN_DIR__/learning/retro-runs.log` — журнал прогонов: `дата \t период \t сессий просканировано`. Append-only, ведёт этот скилл
- `__BRAIN_DIR__/rules/` — существующие правила (для аудита)

## Шаги

### 1. Определить период и собрать сессии

- Период: от даты последнего прогона из `retro-runs.log` до сегодня; лога нет → последние 14 дней; пользователь указал период в аргументах → его.
- Собрать транскрипты: `find ~/.claude/projects -name '*.jsonl' -newermt <начало периода>` + сверить с `__BRAIN_DIR__/inbox/sessions.log` (cwd сессий — для привязки к задачам).
- Отфильтровать: < 20 КБ — пропустить; транскрипт текущей сессии — пропустить.
- Прочитать текущие `patterns.md` (существующие наблюдения) и список правил `ls __BRAIN_DIR__/rules/` — понадобятся для сверки.

### 2. Извлечь сигналы из транскриптов (сабагенты)

Транскрипты в основной контекст не читать. Запускать сабагентов параллельно, каждому — **точные пути** 1–3 транскриптов. Сабагент возвращает по каждой сессии:

**a) Повторяющиеся ручные действия** — цепочки из 2+ шагов, которые агент собирал руками (последовательности команд, однотипные многоходовки «прочитай X → разбери Y → найди Z»). Для каждой: краткое имя, шаги, контекст.

**b) Коррекции пользователя** — маркеры: «опять», «я же говорил», «сколько раз», «почему ты», «не делай», «перестань», tool-rejection с упрёком, переписывание пользователем результата агента. Для каждой: цитата (коротко), в чём была проблема, исправился ли агент, **зафиксировал ли урок в той же сессии** (искать запись в rules/ или упоминание фиксации).

**c) Повторные вопросы пользователю** — фактические вопросы («какая ссылка у...», «как называется...»), ответ на которые мог бы лежать в knowledge/rules.

**d) Срабатывания правил** — какие правила из `rules/` агент в сессии явно применил, процитировал или назвал ([[имя]] в ответах, «по правилу X»). Список имён, без подробностей.

Сабагенту: только чтение; вернуть пусто по сессии без сигналов.

### 3. Агрегация (главный агент)

1. **Слить похожие наблюдения** между сессиями: одинаковая цепочка действий в 3 сессиях = 1 паттерн с 3 наблюдениями. Слияние по сути действия, не по дословному совпадению.
2. **Сверить с `patterns.md`**: паттерн уже есть → увеличить счётчик (добавить наблюдения с session_id и датами); новый → новая запись. Наблюдение без ссылки на конкретную сессию не записывается (анти-фабрикация).
3. **Порог**: паттерн достиг ≥3 наблюдений → черновик proposal (что наблюдалось со ссылками → что предлагается: скилл/правило/рецепт → цена внедрения). Перед предложением сверить с `proposals.md`: отклонённое пользователем не предлагать повторно без новых наблюдений (статус «отклонён» с причиной — память прогонов).
4. **Аудит обучения** по коррекциям из (b):
   - коррекция без зафиксированного правила → секция «незафиксированные уроки» (с предложением правила);
   - коррекция по теме **существующего** правила → повторное нарушение: предложить дописать дату в `violations:` frontmatter правила; при ≥2 нарушениях правило признаётся неработающим в текстовой форме → предложить НЕ дописывание текста, а перенос enforcement ближе к действию (хук, шаг скилла, перенос в rules-ondemand с явным триггером) либо переформулировку.
5. **Жизненный цикл правил** — таблица по `rules/*.md`: срабатывания за период (из (d)) / violations / «тихое» (ни срабатываний, ни нарушений). Тихие два прогона retro подряд (сверить с прошлым отчётом по `retro-runs.log`) → кандидаты на демоушен: в `rules-ondemand/`, в `knowledge/`, в архив — предложить пользователю, не демоутить молча.
6. **Бюджет автозагрузки** — `du -sb` суммы always-loaded части (`rules/*.md`): ориентир ≤150 КБ. Превышен → в отчёт обязательную секцию «кандидаты на демоушен/перенос» (бюджет — триггер консолидации, а не молчаливый рост).
7. **Линт целостности** — запустить `__BRAIN_DIR__/.claude/scripts/lint-brain.sh` (битые wikilinks, пути INDEX.md, ondemand-таблица); находки → в отчёт.

### 4. Показать перед записью

Отчёт пользователю:

- **Незафиксированные уроки** — самое важное, первым: коррекция → предлагаемое правило
- **Повторные нарушения правил** — правило → даты нарушений → предложение (violations / хук / перенос / переформулировка)
- **Паттерны** — таблица: паттерн → счётчик (был/стало) → сессии
- **Proposals** — достигшие порога, с ценой внедрения
- **Жизненный цикл правил** — срабатывания / тихие / кандидаты на демоушен; бюджет автозагрузки (размер, относительно ориентира)

**Отчёт и вопрос — разными ходами** ([[rules/overview-before-question]]): сначала выведи полный отчёт с текстами предлагаемых записей и ОСТАНОВИСЬ — пользователь должен прочитать. AskUserQuestion (что записывать: наблюдения / proposals / правила, multiSelect) — только следующим ходом. Новые правила и переформулировки — **только** после явного одобрения формулировки.

### 5. Записать

После подтверждения:

1. Обновить `patterns.md` (наблюдения, счётчики), `proposals.md` (одобренные к рассмотрению, статус «ждёт решения»).
2. Одобренные правила → `rules/<topic>.md` (Why + How to apply, ссылка на инцидент-сессию).
3. Одобренные violations → frontmatter соответствующих правил.
4. Дописать строку в `retro-runs.log`.
5. Закоммитить и запушить: `cd __BRAIN_DIR__ && git add -A && git commit -m "retro: <дата>" && git push` (пуш — если настроен remote; [[rules/brain-backup]]).

## Запреты

- Не читать транскрипты в основной контекст — только сабагенты.
- Не создавать паттерн/наблюдение без ссылки на конкретную сессию (дата + session_id или путь транскрипта).
- Не внедрять proposals (не создавать скиллы, не менять чужие конфиги) — retro только предлагает; внедрение — отдельная задача после одобрения.
- Не записывать правило без показа формулировки пользователю.
- Не редактировать прошлые наблюдения в patterns.md — только дописывать новые и обновлять счётчики.
