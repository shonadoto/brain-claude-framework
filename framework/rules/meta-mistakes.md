# Mistake-learning rule — фиксировать каждую коррекцию

**Правило:** Каждый раз, когда пользователь корректирует меня, прерывает tool с упрёком, или напоминает правило, которое я уже нарушил («ты почему опять...», «я же говорил», «stop doing X», «не делай Y», tool-rejection + уточняющее сообщение), я ДОЛЖЕН персистить урок в том же turn. Не «в следующий раз буду аккуратнее» — а записанное правило.

**Why:** In-conversation извинения не переживают границы сессии. Без письменной фиксации следующая сессия повторит ту же ошибку. Это meta-правило, которое делает приобретение правил самовосстанавливающимся.

## Обязательные шаги при каждой коррекции

1. **Классифицировать scope** урока:
   - сквозное правило поведения → `__BRAIN_DIR__/rules/<topic>.md`
   - правило с узким триггером («когда делаю X») → `__BRAIN_DIR__/rules-ondemand/<topic>.md` + строка в `rules/ondemand-rules.md`
   - знание «как устроено X» (не правило поведения) → `__BRAIN_DIR__/knowledge/<topic>.md`
   - личное предпочтение → секция «Профиль пользователя» в `__BRAIN_DIR__/.claude/CLAUDE.md`

2. **Написать правило явно** — с тремя частями: rule, **Why:** (инцидент/причина, которую пользователь указал), **How to apply:** (когда/где это срабатывает). Никаких размытых «be careful» — указывай конкретное запрещённое действие и правильную альтернативу.

3. **Сделать это ДО возврата к задаче**, которая была прервана. Persist first, resume after. Если tool был rejected — persist step это часть recovery, не optional, не deferred.

4. **Не говори просто «понял» или «учту»** — покажи пользователю путь записанного файла и формулировку.

## Triggers (non-exhaustive)

- «опять», «я же говорил», «почему ты», «не делай», «перестань», «сколько раз»
- tool rejection с объясняющим сообщением
- пользователь откатывает / переписывает мой результат с комментарием о причине

## Когда НЕ сохранять

Разовые стилевые пожелания только к текущему ответу («короче», «другим тоном»). Только durable-правила.

## Повторное нарушение

Ругань по теме, на которую правило уже есть → дописать дату в frontmatter-поле `violations:` этого правила + рассмотреть переформулировку: правило, которое нарушается повторно, в текущей форме не работает.
