# CONVENTIONS — правила для агентов, работающих с brain

brain — единое личное хранилище знаний и контекстов для работы с Claude. Все агенты, пишущие сюда, обязаны соблюдать эти правила. Скиллы (`inbox-process`, `digest`, `retro`) инлайнят критичные пункты, но source of truth — этот файл.

## Где живёт brain

- Хранилище — директория `__BRAIN_DIR__` (этот файл лежит в её корне).
- VCS — **git** (если настроен): `cd __BRAIN_DIR__ && git add -A && git commit -m "..." && git push` ([[rules/brain-backup]]). Пуш в приватный remote = бэкап.
- Remote (если есть) должен быть **приватным** — brain содержит личный рабочий контекст.

## Структура

```
brain/
├── inbox/                  # сырые заметки (daily notes), inbox/archive/ — обработанные
├── tasks/<task>/           # контекст задач: journal.md (append-only) + CONTEXT.md (дистиллят)
│   └── archive/<task>/     # закрытые задачи — /status туда не ходит
├── projects/<id>/          # контекст проектов (несколько связанных задач)
│   └── archive/<id>/       # завершённые проекты
├── knowledge/<topic>.md    # проверенные знания, не привязанные к задаче
├── learning/               # patterns.md (наблюдения) + proposals.md (предложения)
├── loops.md                # открытые петли вне задач: сделать / жду / вопросы / потом
├── rules/                  # правила Claude, автозагружаются в каждую сессию
├── rules-ondemand/         # правила «по требованию» — читаются при входе в тему
├── templates/              # шаблоны новых сущностей
├── INDEX.md                # карта сущностей, 1 строка на сущность
└── .claude/                # CLAUDE.md, skills, hooks
```

**Что такое задача.** Задача — любая именованная рабочая активность: отчёт, сделка, разбор, переговоры, фича. Имя — короткий kebab-case-слаг (`otchet-q2-2026`, `sdelka-alfa`) или ID из внешней системы, если она есть.

**Граница task / project:**

- Несколько связанных задач с общей целью — проект (`projects/<id>/`); одиночная задача — `tasks/<task>/`.
- **Связь** — frontmatter задачи: `project: <id>`. Журналы всех задач живут в `tasks/<task>/journal.md` — скиллы пишут туда как обычно.
- **Один факт — один дом (анти-дубль):** решения одной задачи — journal/CONTEXT этой задачи; решения шире одной задачи (договорённости, схема, роли) — проект; не копировать — ссылаться `[[...]]`.
- **Триггер обновления проекта:** значимый этап по задаче с `project:` (этап завершён / задача закрыта) → в той же сессии обновить строку задачи в таблице проекта + дату снапшота.
- **Закрытие → архив:** задача закрыта (результат подтверждён, либо пользователь сказал «закрыто») → frontmatter `status: archived`, директория переезжает в `tasks/archive/<task>/`, строка INDEX — в секцию «Архив». Завершённый проект — аналогично в `projects/archive/<id>/`. `/status` и прочие сводки архив не обходят; журналы в архиве остаются append-only (дописывать можно, например из /digest).

## Формат — Obsidian-совместимый markdown

- Vault = эта директория. Файлы — markdown с YAML frontmatter.
- Имена файлов — kebab-case, без пробелов. Один файл = одна сущность. H1 = имя сущности.
- Связи — `[[wikilinks]]`: journal линкует `[[projects/<id>]]`, knowledge линкует задачи-источники, правило линкует инцидент.
- Frontmatter (минимум):

```yaml
---
type: task | project | knowledge | decision | rule | pattern
source: <task> | "пользователь, YYYY-MM-DD" | transcript | <документ/ссылка>
created: YYYY-MM-DD
status: active | verified | outdated | archived
tags: []
---
```

## Общие правила (все пишущие агенты)

1. **У каждого факта — источник.** Каждая запись несёт пометку, откуда она: задача, «сказал пользователь (дата)», документ, вывод команды, транскрипт сессии. Без источника — только с явным маркером `гипотеза:`. Через месяц агент будет действовать на основе записи — он должен видеть, на чём она стоит.
2. **Не выдумывать структуру.** Перед записью — карта существующего (`INDEX.md`); раскладывать в существующие сущности. Создание новой сущности/категории — явное действие, отмечается в отчёте с обоснованием.
3. **Решётка записи: ДОПОЛНИТЬ / СОЗДАТЬ / УСТАРЕЛО / НИЧЕГО.** Перед любой записью в rules/knowledge — поиск соседей (grep по ключевым словам, INDEX, wikilinks) и явный выбор операции; default-подозрение — ДОПОЛНИТЬ существующее, не СОЗДАТЬ новое. Новое противоречит существующему → НЕ перезаписывать молча: пометить конфликт («раньше считали X (источник), теперь Y (источник)»), старое — `status: outdated` + «заменено [[Y]]». После записи пройти соседей по wikilinks: проставить обратные ссылки.
4. **Ничего не удалять.** Только archive + лог маршрутизации. Исключение — секреты (п. 8).
5. **Абсолютные даты.** «Вчера», «после встречи» → конкретные даты при обработке. Сырой inbox может говорить «завтра», обработанная запись — нет.
6. **Описывать сущность, а не процесс её добывания.** Не «я проанализировал транскрипт и пришёл к выводу...» — а суть факта. Откуда — в frontmatter `source`, одной строкой.
7. **Язык пользователя.** Те же правила, что для чата ([[rules/language]]). Имена файлов/команд/терминов — as-is.
8. **Никаких секретов.** brain может бэкапиться во внешний git-remote. Токены, пароли, ключи, чужие персональные данные — не пишутся. Заметка их содержит → при обработке вычищать с пометкой `[secret removed]`.
9. **Решения — с «почему».** Решение + причина + (если были) отвергнутые альтернативы одной строкой. Решение без rationale через полгода бесполезно.
10. **Консолидация — только локальными дельтами.** Никогда не переписывать корпус (rules/, knowledge/, длинный файл «для наведения порядка») одним прогоном — только точечные правки с диффом: монолитная LLM-перезапись схлопывает корпус (теряет большую часть содержимого молча). При слиянии/сжатии записей сохранять конкретику — команды, пути, точные формулировки, имена; обобщение до «быть внимательнее» уничтожает ценность записи. Маркеры деградации, запрещённые в записях: «остальное как раньше», «см. выше», «аналогично предыдущему».

## Правила по слоям

### inbox/ — захват

Ноль трения. `/inbox` аппендит строку с таймстампом в `inbox/YYYY-MM-DD.md` — без классификации, без вопросов, не трогая остальной brain.

### tasks/<task>/journal.md — журналы

- **Append-only.** Прошлые записи не редактируются — только дописывание. Вчерашний вывод неверен → сегодня дописать «вывод от <дата> неверен, потому что...».
- Запись = дата + источник (сессия) + что сделано / что решено / что заблокировано. 1–3 строки, не эссе.
- **Писать ПО ХОДУ сессии**, немедленно при событии (не дожидаясь конца):
  1. принято решение (выбран подход, отвергнута альтернатива) — решение + почему
  2. найден тупик («X не работает из-за Y») — тупики обязательны, их агенты повторяют чаще всего
  3. получен внешний факт (коллега сказал, пользователь уточнил, выяснилось из документа)
  4. завершён значимый этап (план готов, документ отправлен, ответ получен)
- Рутинные действия (прочитал файл, выполнил поиск) — НЕ журналируются. Journal ≠ transcript.

### tasks/<task>/CONTEXT.md, projects/<id>/CONTEXT.md — дистилляты

- **Перегенерируемые.** Не содержат ничего, чего нет в journal/artifacts — это view, не storage.
- **Бюджет на размер:** ~экран-полтора. CONTEXT.md читается агентом в начале каждой сессии — это дорогое контекстное окно. Детали — ссылкой в journal.
- Фиксированная структура: цель → текущий статус → ключевые решения → открытые вопросы → грабли. Шаблон — `templates/task-context.md`.

### knowledge/ — знания

- **Только подтверждённые факты.** Гипотезы живут в journal с маркером `гипотеза:`. Гипотеза подтвердилась → переносится в knowledge с источником подтверждения. Frontmatter `status: verified` обязателен.
- **Не дублировать внешние системы записи.** Что выводимо из документов, систем, переписок — не хранится, хранится указатель. brain хранит то, чего больше нигде нет: устные договорённости, выводы из инцидентов, «как на самом деле работает X».
- **Выяснил «где живёт X» — запиши в той же сессии.** Ресерч «где на самом деле лежит документ/настройка/ответственный» стоил усилий — результат обязан осесть в knowledge/ сразу, не дожидаясь digest.
- Снапшоты внешних данных и протухающие факты (статусы, версии, договорённости «на сейчас») — с **двумя датами**: «записано YYYY-MM-DD» и «проверено/верно на YYYY-MM-DD» (вторая обновляется при каждой перепроверке). Читающий агент по возрасту второй даты решает, перепроверять ли перед действием ([[rules/verify-before-acting]]). Устаревший факт не удалять и не переписывать молча — пометить «устарело с <даты>, заменено [[X]]».
- **Оси knowledge** (подкаталоги по темам) появляются по мере роста. Новая ось — явное действие с обоснованием в отчёте (правило «не выдумывать структуру»).
- **Индекс один — `INDEX.md`.** Секционные оглавления внутри подкаталогов не заводим: они дублируют INDEX и расходятся с ним. При добавлении файла знания — строка в `INDEX.md`, секция «Знания».

### loops.md — открытые петли

- Незавершённое **вне задач**: действия, ожидания от других, открытые вопросы, отложенные идеи. Хвосты по задачам живут в `tasks/<task>/CONTEXT.md` — в loops не дублируются.
- **Живой файл** (исключение из «append-only»): закрытая петля чекается `[x]` с датой; разросшиеся закрытые переносятся в `loops-archive.md` с подтверждения пользователя.
- Строка петли: дата + суть + источник; «Жду» — обязательно от кого/чего.
- Захват — триггерные слова пользователя («не забыть», «надо будет потом», «жду от», «спросить у») → агент дописывает строку сразу, в той же сессии. Закрытие — только по словам пользователя или явному факту (ответ получен — с источником).
- Сводку показывает `/status`.

### learning/ — самообучение

- **Паттерн ≠ правило.** `patterns.md` — наблюдения со счётчиками и ссылками на конкретные случаи (транскрипт/дата). При ≥3 наблюдениях → `proposals.md`. Правилом/скиллом становится **только после одобрения пользователя**.
- Каждый proposal: что наблюдалось (со ссылками) → что предлагается → цена внедрения. Без ссылок на конкретные случаи proposal не создаётся.

### rules/ — правила Claude

- Автозагружаются в каждую сессию (симлинк `~/.claude/rules/` → `brain/rules/`).
- Формат правила: rule + **Why** (конкретный инцидент) + **How to apply** (когда/где срабатывает). Никаких размытых «be careful». Шаблон — `templates/rule.md`.
- Правило — тоже Obsidian-нота: линкует инцидент в journal, из которого родилось.
- Правила с узким триггером («когда делаю X») — в `rules-ondemand/` + строка с триггером в `rules/ondemand-rules.md` (чтобы не раздувать автозагрузку).

## Контур обучения (коррекции пользователя)

Когда пользователь корректирует агента, прерывает tool с упрёком или напоминает уже нарушенное правило — в том же turn:

1. **Диагноз с подтверждением.** Агент формулирует понимание: «Понял так: проблема в X, нужно было Y. Фиксирую правило: ...». Причина неочевидна → ОДНА уточняющая строка, не угадывание. Записать неправильный урок хуже, чем никакой.
2. **Записать правило** в `rules/` (Why + How to apply), показать пользователю путь файла и формулировку.
3. **Persist first, resume after** — фиксация до возврата к прерванной задаче.
4. Ругань на тему, по которой правило уже есть → дописать дату в frontmatter-поле `violations: []` этого правила + рассмотреть переформулировку (правило не держит).

`/retro` аудирует контур: скан транскриптов на маркеры коррекций («опять», «я же говорил», «сколько раз», tool-rejection с упрёком) → проверка, что в той же сессии появилась запись. Не появилась → флаг в отчёте.

## Правила сортировщика (/inbox-process)

1. **Сомнение → вопрос, не угадывание.** Неоднозначные заметки копятся и задаются одной пачкой (AskUserQuestion) в конце прогона.
2. **Отчёт обязателен:** таблица «заметка → куда → почему» за прогон.
3. **Идемпотентность.** Обработанные заметки физически переносятся в `inbox/archive/` с пометкой маршрута. Повторный запуск не плодит дубли.
4. **Внешние действия — только с подтверждением.** «Надо написать X» → предложить, не писать. То же для правок вне brain.
5. **Заметка может разойтись в несколько мест** (journal задачи + learning/patterns) — оба маршрута в отчёте.
