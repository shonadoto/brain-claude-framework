# /retro — ретроспектива поперёк сессий

Корень brain (далее `$BRAIN`) — содержимое файла `~/.claude/brain-dir`; файла нет → `~/brain`.

Анализирует транскрипты за период и отвечает на вопрос «**как** работает агент», а не «что делалось» (это `/digest`). Два контура: майнинг паттернов и аудит обучения. Правила — `$BRAIN/CONVENTIONS.md`, разделы «learning/» и «Контур обучения».

## Файлы

- `~/.claude/projects/*/` — транскрипты сессий (`*.jsonl`)
- `$BRAIN/learning/patterns.md` — наблюдения со счётчиками (пополняет retro и digest)
- `$BRAIN/learning/proposals.md` — предложения, достигшие порога ≥3 наблюдений
- `$BRAIN/learning/retro-runs.log` — журнал прогонов: `дата \t период \t сессий просканировано`. Append-only, ведёт этот скилл
- `$BRAIN/rules/` — существующие правила (для аудита)

## Шаги

### 1. Определить период и собрать сессии

- Период: от даты последнего прогона из `retro-runs.log` до сегодня; лога нет → последние 14 дней; пользователь указал период в аргументах → его.
- Собрать транскрипты: `find ~/.claude/projects -name '*.jsonl' -newermt <начало периода>` + сверить с `$BRAIN/inbox/sessions.log` (cwd сессий — для привязки к задачам).
- Отфильтровать: < 20 КБ — пропустить; транскрипт текущей сессии — пропустить.
- Прочитать текущие `patterns.md` (существующие наблюдения) и список правил `ls $BRAIN/rules/` — понадобятся для сверки.

### 2. Извлечь сигналы из транскриптов (сабагенты)

Транскрипты в основной контекст не читать. Запускать сабагентов параллельно, каждому — **точные пути** 1–3 транскриптов. Сабагент возвращает по каждой сессии:

**a) Повторяющиеся ручные действия** — цепочки из 2+ шагов, которые агент собирал руками (последовательности команд, однотипные многоходовки «прочитай X → разбери Y → найди Z»). Для каждой: краткое имя, шаги, контекст.

**b) Коррекции пользователя** — маркеры: «опять», «я же говорил», «сколько раз», «почему ты», «не делай», «перестань», tool-rejection с упрёком, переписывание пользователем результата агента. Для каждой: цитата (коротко), в чём была проблема, исправился ли агент, **зафиксировал ли урок в той же сессии** (искать запись в rules/ или упоминание фиксации).

**c) Повторные вопросы пользователю** — фактические вопросы («какая ссылка у...», «как называется...»), ответ на которые мог бы лежать в knowledge/rules.

**d) Срабатывания правил** — какие правила из `rules/` агент в сессии явно применил, процитировал или назвал ([[имя]] в ответах, «по правилу X»). Список имён, без подробностей.

**e) Подтверждения пользователя** — явные одобрения приёма («вот так правильно», «отлично, так и делай всегда», «да, в таком формате»). Для каждого: что именно одобрено, цитата коротко. Подтверждённый приём — такой же сигнал обучения, как коррекция, только со знаком плюс.

Сабагенту: только чтение; вернуть пусто по сессии без сигналов.

### 3. Агрегация (главный агент)

1. **Слить похожие наблюдения** между сессиями: одинаковая цепочка действий в 3 сессиях = 1 паттерн с 3 наблюдениями. Слияние по сути действия, не по дословному совпадению.
2. **Сверить с `patterns.md`**: паттерн уже есть → увеличить счётчик (добавить наблюдения с session_id и датами); новый → новая запись. Наблюдение без ссылки на конкретную сессию не записывается (анти-фабрикация).
3. **Порог**: паттерн достиг ≥3 наблюдений → черновик proposal (что наблюдалось со ссылками → что предлагается: скилл/правило/рецепт → цена внедрения). Перед предложением сверить с `proposals.md`: отклонённое пользователем не предлагать повторно без новых наблюдений (статус «отклонён» с причиной — память прогонов).
4. **Аудит обучения** по коррекциям из (b):
   - коррекция без зафиксированного правила → секция «незафиксированные уроки» (с предложением правила);
   - коррекция по теме **существующего** правила → повторное нарушение: предложить дату в `violations:` frontmatter И **усиление правила** — не просто дописать дату, а сделать правило жёстче: переформулировать категоричнее (НИКОГДА/ВСЕГДА в первой строке), конкретизировать How to apply под пойманный случай, поднять выше в файле. Показать пользователю before/after формулировки. При ≥2 нарушениях текстовая форма признаётся неработающей → предложить перенос enforcement ближе к действию (хук, шаг скилла, перенос в rules-ondemand с явным триггером).
5. **Жизненный цикл правил** — таблица по `rules/*.md`: срабатывания за период (из (d)) / **коррекций по теме правила за период и тренд** (в прошлом периоде 2 → сейчас 0 = усваивается; растёт = правило не работает) / «тихое» (ни срабатываний, ни нарушений). Тихие два прогона retro подряд (сверить с прошлым отчётом по `retro-runs.log`) → кандидаты на демоушен: в `rules-ondemand/`, в `knowledge/`, в архив — предложить пользователю, не демоутить молча.
6. **Бюджет автозагрузки** — `du -sb` суммы always-loaded части (`rules/*.md`): ориентир ≤150 КБ. Превышен → в отчёт обязательную секцию «кандидаты на демоушен/перенос» (бюджет — триггер консолидации, а не молчаливый рост).
7. **Линт целостности** — запустить `$BRAIN/.claude/scripts/lint-brain.sh` (битые wikilinks, пути INDEX.md, ondemand-таблица, знания-сироты и без дат проверки); находки → в отчёт.
8. **Протухание знаний** — взять 3–5 файлов `knowledge/` с самой старой датой «проверено/верно на» (или вовсе без неё): проверить упомянутые в них артефакты (файлы, пути, договорённости — что проверяемо дёшево); подтверждается → обновить дату проверки; нет → предложить пометку «устарело с <даты>». Не переписывать факты молча.

### 4. Показать перед записью

Отчёт пользователю:

- **Незафиксированные уроки** — самое важное, первым: коррекция → предлагаемое правило
- **Повторные нарушения правил** — правило → даты нарушений → усиленная формулировка before/after (или хук / перенос)
- **Подтверждённые приёмы** — одобрения из (e): что закрепить как паттерн
- **Паттерны** — таблица: паттерн → счётчик (был/стало) → сессии
- **Proposals** — достигшие порога, с ценой внедрения
- **Жизненный цикл правил** — срабатывания / тренд коррекций / тихие / кандидаты на демоушен; бюджет автозагрузки (размер, относительно ориентира)
- **Протухшие знания** — файл → что не подтвердилось → предложение

**Отчёт и вопрос — разными ходами** ([[rules/overview-before-question]]): сначала выведи полный отчёт с текстами предлагаемых записей и ОСТАНОВИСЬ — пользователь должен прочитать. AskUserQuestion (что записывать: наблюдения / proposals / правила, multiSelect) — только следующим ходом. Новые правила и переформулировки — **только** после явного одобрения формулировки.

### 5. Записать

После подтверждения:

1. Обновить `patterns.md` (наблюдения, счётчики; подтверждённые пользователем приёмы — с пометкой «одобрено пользователем, <дата>»), `proposals.md` (одобренные к рассмотрению, статус «ждёт решения»).
2. Одобренные правила → `rules/<topic>.md` (Why + How to apply, ссылка на инцидент-сессию). Перед записью — чеклист качества правила (CONVENTIONS, «rules/»).
3. Одобренные violations и усиленные формулировки → соответствующие правила.
4. Дописать строку в `retro-runs.log`.
5. Закоммитить и запушить: `cd $BRAIN && git add -A && git commit -m "retro: <дата>" && git push` (пуш — если настроен remote; [[rules/brain-backup]]).

## Запреты

- Не читать транскрипты в основной контекст — только сабагенты.
- Не создавать паттерн/наблюдение без ссылки на конкретную сессию (дата + session_id или путь транскрипта).
- Не внедрять proposals (не создавать скиллы, не менять чужие конфиги) — retro только предлагает; внедрение — отдельная задача после одобрения.
- Не записывать правило без показа формулировки пользователю.
- Не редактировать прошлые наблюдения в patterns.md — только дописывать новые и обновлять счётчики.
