# /setup-brain — первичная настройка

Корень brain (далее `$BRAIN`) — содержимое файла `~/.claude/brain-dir`; файла нет → `~/brain`.

Onboarding-интервью после установки фреймворка. Цель: заполнить профиль пользователя, понять его рабочий контекст и завести первые сущности. Дружелюбно и коротко — пользователь может быть не-программистом; никакого жаргона.

## Шаги

### 1. Проверка окружения

Быстро и молча проверить:

- `$BRAIN/CONVENTIONS.md` существует;
- `~/.claude/rules` — симлинк на `$BRAIN/rules`;
- `cd $BRAIN && git rev-parse --is-inside-work-tree` — git настроен или нет (не ошибка, просто факт для шага 4).

Что-то сломано → сказать пользователю простыми словами, что и как починить (или предложить перезапустить `install.sh`).

### 2. Интервью (AskUserQuestion, по 1-2 вопроса за ход)

Спросить по очереди, с вариантами там, где они уместны:

1. Как к тебе обращаться (имя)?
2. Чем ты занимаешься? (сфера, роль — свободный текст; это определит, какие задачи и термины будут в brain)
3. На каком языке общаемся? (по умолчанию русский)
4. Какие у тебя сейчас 1-3 главные рабочие активности? (станут первыми задачами в `tasks/`)
5. Есть ли постоянные зоны ответственности без «конца» — клиенты, команда, здоровье, семейные финансы? (станут зонами в `areas/`)
6. Есть ли что-то, что Claude должен всегда делать или никогда не делать? (станет первыми личными правилами)

Не допрашивать: пользователь отвечает коротко или «пропусти» → идти дальше.

### 3. Записать

1. Заполнить секцию «Профиль пользователя» в `$BRAIN/.claude/CLAUDE.md` (имя, занятие, язык, предпочтения).
2. Для каждой названной активности: создать `tasks/<slug>/CONTEXT.md` по шаблону `templates/task-context.md` (цель со слов пользователя, этап «начато») + пустой `journal.md` + строка в `INDEX.md`.
2a. Для каждой названной зоны: `areas/<slug>/CONTEXT.md` по шаблону `templates/area-context.md` + строка в `INDEX.md`, секция «Зоны».
3. Названные «всегда/никогда» → правила в `rules/<topic>.md` по шаблону `templates/rule.md` (Why: «пользователь, <дата>, при настройке»). Показать пути и формулировки.
4. Если git настроен: `cd $BRAIN && git add -A && git commit -m "setup: первичная настройка" && git push` (пуш — если есть remote).

### 4. Показать, что дальше

Короткая шпаргалка (адаптировать под ответы из интервью, без жаргона):

- «пришла мысль/задача — скажи `/inbox <текст>`, разберём потом»
- «`/status` — покажу всё незавершённое»
- «`/inbox-process` — разобрать накопленные заметки»
- «`/plan-day` утром и `/close-day` вечером — план дня из хвостов и его закрытие»
- «`/digest` — время от времени: вытащить важное из прошлых разговоров»
- «`/retro` — раз в неделю-две: чему я научился и что улучшить»
- «`/brain-doctor` — если что-то не работает; `/update-brain` — обновить фреймворк (я подскажу, когда выйдет новая версия)»
- «поправил меня — я записываю урок в правила и больше так не делаю»
- если git не настроен — одной строкой: «бэкап не настроен; захочешь — см. README, раздел "Бэкап"».

## Запреты

- Не создавать сущности, которые пользователь не называл ([[rules/no-overengineering]]).
- Не пропускать показ записанных правил (пути + формулировки).
